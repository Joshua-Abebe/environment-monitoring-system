Design Choices

Date 05/11/2026

- I chose to build a simple-realistic multi-room monitoring system, because it will allow me:
               - To learn:
                           - Event-driven systems
                           - Data orchestration across different database systems according to their strengths
               - To finish project on time while not compromising learning objectives

- I chose these five analysis for a start for the system to perform:
               - Latest sensor readings
               - Average temprature per room
               - Sensor network visualization
               - Environmental ris* alerts
               - Raw packet history (for debugging, replay, anomaly detection, and etc)

- Why I chose MQTT as the communication protocol aside from my Professor recommendation is because:
               - It is a light weight protocol
               - It is open source
               - It is easy to implement
               - It is easy to scale
               -

- I chose these three database for the following reasons according to my project:
               - MySQL
                      - Very useful in statistical analysis
                      - Very fast and reliable in math queries
               - MongoDB
                      - For flexibility (if the sensors add other metrics and information, this database don't need any additional alteration to implement the change)
                      - For historical data
                      - To get the raw data from the sensors
               - Neo4j
                      - Unmatched in relationship's traversal and query performance
                      - For visualizaiton (specifically to visualize sensor network)

- I chose Python as the programming language, because:
               - It intergrates well with the database
               - It is easier to simulate the sensor's data
               - It is easy to integrate with MQTT according to online searches
               - It doesn't have too much boilerplate code
               - It is easy to implement





Date 05/12/2026

- This is my MQTT topic design, simple and descriptive enough to show the hierchacy of environment monitoring system:
               - building/ {Location} / {Sensor_type}

- I chose to have only one MQTT subscriber to manage all three databases as opposed to having different subscribers for each databases to reduce the complexity
               - MQTT Subscriber
                     ├── MySQL
                     ├── MongoDB
                     └── Neo4j

- I chose to have ONE SENSOR = ONE METRIC, because:
               - To make the relationship meaningful(especially with Neo4j)
               - To reduce the complexity of the system, especially for the MQTT subscriber (it will be flooded with messages which wil not be easy to handle)
                            - Cleaner topics
               - To reduce the complecity of the database (for MySQL, har*er to query), which makes it easier to scale later

Date 05/13/2026

- This is MQTT life cycle that I've decided on:

Sensor Simulator (all sensors generate events asynchronously)
      ↓
 MQTT Publish (only know about broker's address and topic)
      ↓
 Mosquitto Broker (doesn't understand meaning of the data nor store data permanently)
      ↓
 Subscriber Engine (writes directly to database)
      ↓
 Validation Layer (ensuring the incoming message is not distorted, truncated, or extreme)
      ↓
 Processing Layer (after validation the data becomes trusted)
      ↓
 Database Routing (the message will be stored)
      ↓
 Alert Engine (after the event gets stored, evaluate ris* and send alerts)
      ↓
 Analytics System (querying and insight generation)



- The reason for the decision on the Subscriber Engine to write directly to database came from the idea not to make it complicated with the first version by creating queues and other layers to the incoming data, but I might implement in the future
- In the Validation Layer, we ensure the incoming message have these:
                   - sensor_id
                   - value
                   - timestamp
        With value range checks:
                    - Temperature: -20 to 80
                    - Humidity: 0% to 100%
                    - AQI: 0 to 500
- And any invalid message will be logged
- In the Processing Layer, the message will be sorted and stored according to the database design and rules

THIS IS WHAT EACH OF THE SYSTEM INITIALIZATION STAGES ARE ABOUT:
Load configuration:
- Use .env file

Connect databases:
- Separate database logic for all of them
- Each database get its own handler module
               database/
                    |--mysql_handler.py
                    |--mongodb_handler.py
                    |--neo4j_handler.py

MySQL tables initialization:
- When app starts, these table should exist:
                    - Rooms (autogenerated at startup)
                    - Sensors (autogenerated at startup)
                    - Readings


Neo4j graph initialization:
- Initializes only once at a startup, mostly stable

"Topology is static, measurement is dynamic"


MQTT subscriber startup:
- It should run forever


Sensor simulation engine:
- Each sensor run independently using their own threads to make them work asynchronously, reason:
                    - Sensors become decouples from one another
                    - Each sensor can publish within its own time frame
                    - Each sensor behaves autonomosly


Analytics engine:
- The read/query layer, separated from ingestion logic
                    - analytics/
                           |--analytics_engine.py
                           |--mysql_queries.py
                           |--mongodb_queries.py
                           |--neo4j_queries.py

 Then I dicided which entities should be counted as entities:
 Object modeling:
 CLASS 1 - Sensor:
               - Should know its id
               - Should know its metric
               - Should know its location
               - Should be able to generate values
               - Should be able to send MQTT messages

               Sensor
                ├── sensor_id
                ├── sensor_type
                ├── location
                ├── unit
                ├── current_value
                ├── generate_value()
                └── publish()

 CLASS 2 - SensorManager:
               - Creates sensor
               - Starts threads
               - Coordinates publishing

               keeping orchestration separate from sensor behavior

 CLASS 3 - EventProcessor:
               - Receive MQTT messages
               - Validates events
               - Route to databases
               - Trig**r ale**s
